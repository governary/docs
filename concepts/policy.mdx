Governix policy is a set of tenant-scoped rules that the control plane evaluates before every AWS Bedrock call. Instead of enforcing access controls at the application layer — where each team builds their own checks — you define what each tenant is allowed to do once, and Governix enforces it consistently across every request, regardless of which application sent it. This page explains what policy controls, when it runs, and how to think about the two integration modes.

## What policy controls

A tenant policy defines the boundaries of what that tenant's applications can do when calling Bedrock. You configure rules across five areas:

<AccordionGroup>
  <Accordion title="Allowed knowledge bases">
    Specify which Bedrock Knowledge Base IDs a tenant's applications may retrieve from. Requests that attempt to query an unlisted knowledge base are denied before the retrieval call is made.
  </Accordion>
  <Accordion title="Allowed models">
    List the Bedrock model IDs the tenant is permitted to invoke. If an application requests a model outside this list, Governix can deny the request or downgrade it to an approved model.
  </Accordion>
  <Accordion title="Citation requirements">
    Require that responses include source citations from retrieved chunks. This is useful for regulated use cases where answers must be traceable to a specific document or passage.
  </Accordion>
  <Accordion title="Fallback rules">
    Define what Governix should do when a primary knowledge base or model is unavailable or denied. Fallback rules prevent silent failures and keep applications predictable.
  </Accordion>
  <Accordion title="Quota thresholds">
    Set a request or token ceiling for the tenant. When usage reaches the threshold, Governix can raise a Warning status or trigger a quota-block action on subsequent requests.
  </Accordion>
</AccordionGroup>

## Runtime actions

When Governix evaluates a request against the tenant's policy, it selects one of five actions:

| Action | What happens |
|---|---|
| **Allow** | The request passes through to Bedrock unchanged. |
| **Deny** | The request is rejected and a denial event is written to the audit ledger. The Bedrock call never happens. |
| **Force filter** | Governix applies a metadata filter to the retrieval query before it reaches the knowledge base, restricting which chunks the model can see. |
| **Downgrade model** | The requested model is replaced with a permitted alternative. The request continues with the substituted model. |
| **Quota block** | The tenant has exceeded its configured threshold. The request is rejected and a `quota_block` event is written to the audit ledger. |

<Warning>
  Deny and quota-block actions are terminal — the Bedrock call does not proceed. Make sure your application handles these error responses gracefully rather than surfacing raw policy errors to end users.
</Warning>

## When policy is evaluated

Policy evaluation happens synchronously, before the Bedrock call is dispatched. The sequence is:

<Steps>
  <Step title="Request arrives at the control plane">
    Your application sends a request to the Governix control plane using its registered API key. Governix identifies the application and its parent tenant.
  </Step>
  <Step title="Policy engine evaluates the request">
    Governix checks the tenant's policy rules against the request parameters — requested model, target knowledge base, current quota usage — and selects the appropriate runtime action.
  </Step>
  <Step title="Action is applied">
    The selected action is executed. If the action is Allow, Force filter, or Downgrade model, the (possibly modified) request is forwarded to Bedrock. If the action is Deny or Quota block, the request is rejected and an audit event is written.
  </Step>
  <Step title="Response is returned">
    Bedrock returns a response, Governix records the completion event in the audit ledger, and the response is returned to your application.
  </Step>
</Steps>

## Integration modes

Governix supports two integration modes that reflect how your application structures its Bedrock calls. Policy evaluation applies in both modes, but the shape of the request differs.

<Tabs>
  <Tab title="Mode A — RetrieveAndGenerate">
    In Mode A, your application calls Bedrock's `RetrieveAndGenerate` API, which handles both the knowledge base retrieval and the model generation in a single request. Before making that call, your application calls the Governix policy evaluation API with the knowledge base ID and model ID, and Governix returns the appropriate action to apply.

    This mode is the simpler integration path. Use it when your application already uses `RetrieveAndGenerate` and you want to add governance without restructuring the call flow.

    <Info>
      Force filter actions in Mode A are applied to the retrieval portion of the `RetrieveAndGenerate` request. Governix injects the required metadata filter before the call reaches the knowledge base.
    </Info>
  </Tab>
  <Tab title="Mode B — Retrieve + Converse">
    In Mode B, your application makes two separate Bedrock calls: a `Retrieve` call to fetch chunks from a knowledge base, followed by a `Converse` call to generate a response using those chunks. Your application calls Governix policy evaluation once before either Bedrock call, then sends separate events after each step completes.

    This mode gives you finer control over the retrieval and generation steps. Use it when your application already separates retrieval from generation, or when you need to inspect or transform retrieved chunks before passing them to the model.

    <Info>
      Governix records separate audit events for the Retrieve and Converse calls in Mode B. Both events are linked by a common request trace ID in the audit ledger.
    </Info>
  </Tab>
</Tabs>

## Policy scope in the current release

<Note>
  In the current Governix release, policy is tenant-scoped by default. Rules apply uniformly to all applications under a tenant. Application-level policy overrides are planned for a future release.
</Note>
