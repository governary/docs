Tenants are the foundational unit of governance in Governix. Each tenant represents a discrete governed boundary — a department, a customer, a business unit, or any other logical grouping you want to control, meter, and audit independently. Every policy rule, usage metric, and audit record in Governix is scoped to a tenant, which means you can enforce rules, track consumption, and export evidence at exactly the right level of granularity.

## What a tenant represents

A tenant maps to whatever boundary makes sense for your organization. Common examples include:

- A customer account in a SaaS product
- An internal department consuming shared AI infrastructure
- A regulated workload that requires isolated audit trails

<Note>
  Tenants are the top-level unit of governance in Governix. All applications, policy rules, usage metrics, and audit records belong to a tenant.
</Note>

## What Governix tracks per tenant

For each tenant, Governix records a live snapshot of activity and health:

<AccordionGroup>
  <Accordion title="Request count">
    The total number of Bedrock requests attributed to this tenant across all of its applications. This count updates in real time as requests flow through the control plane.
  </Accordion>
  <Accordion title="Token usage">
    Cumulative input and output tokens consumed by the tenant's applications. Token data is broken down by model in the usage view.
  </Accordion>
  <Accordion title="Estimated cost">
    A showback figure derived from token usage and model pricing. Use this to allocate costs back to departments or customers without building a separate billing pipeline.
  </Accordion>
  <Accordion title="Quota percentage">
    How far into the tenant's configured quota threshold the current usage has gone. Governix surfaces a Warning state when a tenant approaches its limit.
  </Accordion>
  <Accordion title="Blocked events">
    A count of requests that the policy engine denied or quota-blocked for this tenant. Review blocked events in the audit ledger to understand why requests failed.
  </Accordion>
</AccordionGroup>

## The admin console tenant list

The admin console at `app.governary.io/tenants` gives you a real-time overview of all tenants. Each row in the tenant list shows:

| Field | Description |
|---|---|
| Name | The human-readable tenant name you assigned at creation |
| Tenant ID | A stable identifier such as `tenant_01` used across policy rules and API keys |
| Apps | The number of registered applications under this tenant |
| Quota % | Current usage as a percentage of the configured quota threshold |
| Requests | Cumulative request count for the current period |
| Status | `Active` when usage is within normal bounds; `Warning` when quota is approaching the threshold |

<Info>
  A tenant showing a Warning status has not yet been blocked — Governix raises the warning state to give you time to adjust the quota or contact the tenant before a hard quota block takes effect.
</Info>

## How tenants relate to applications and policy

A tenant acts as the parent for two other key concepts:

<CardGroup cols={2}>
  <Card title="Applications" icon="grid-2" href="/concepts/applications">
    Each Bedrock RAG system or AI workload registers as an application under a tenant. A single tenant can have multiple applications. Usage and audit records are attributed to both the application and its parent tenant.
  </Card>
  <Card title="Policy" icon="shield-check" href="/concepts/policy">
    Policy in Governix is tenant-scoped by default. You define which knowledge bases and models a tenant is allowed to use, and those rules are enforced at runtime before every Bedrock call.
  </Card>
</CardGroup>
