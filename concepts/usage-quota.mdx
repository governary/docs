---
title: "Usage and quota: track and limit Bedrock consumption"
description: "Understand how Governix tracks requests, tokens, and estimated cost per tenant and application, and how quota thresholds trigger warnings and blocks."
---

Governix meters every request that flows through the control plane and attributes it to the tenant and application that sent it. This gives you a real-time picture of who is consuming your Bedrock capacity, how much it costs, and how close each tenant is to its configured limit. When a tenant approaches or exceeds its quota, Governix surfaces a warning state or blocks further requests — without requiring you to build any metering logic in your applications.

## What Governix tracks

For every request that passes through the control plane, Governix records:

<AccordionGroup>
  <Accordion title="Request count">
    A running total of requests attributed to the tenant and application for the current period. This is the primary signal for understanding activity volume and the simplest way to spot unexpected spikes.
  </Accordion>
  <Accordion title="Token usage">
    Input and output tokens consumed per request, accumulated at the tenant and application level. Token counts are broken down by model in the usage view so you can see which model is driving consumption.
  </Accordion>
  <Accordion title="Estimated cost">
    A showback figure calculated from token usage and the per-token pricing of the invoked model. Use this for internal cost allocation — showing departments or customers what their AI usage is worth — without needing a separate billing pipeline.
  </Accordion>
  <Accordion title="Quota percentage">
    How much of the tenant's configured quota has been consumed in the current period, expressed as a percentage. The admin console displays this figure in the tenant list and surfaces a status badge that changes as usage climbs.
  </Accordion>
</AccordionGroup>

## Quota thresholds and status states

Each tenant has a configurable quota threshold. As usage grows toward that threshold, Governix transitions through two states:

| Status | Condition | What it means |
|---|---|---|
| **Active** | Usage is below the warning threshold | The tenant is operating normally. Requests are evaluated by policy and forwarded to Bedrock. |
| **Warning** | Usage is approaching the threshold (for example, 89%) | Governix raises a visible warning in the admin console. Requests still proceed, but you should review the tenant's quota before it is exhausted. |

<Note>
  A Warning status does not block requests on its own. It is a signal for you to act — either by raising the quota, reaching out to the tenant, or letting the quota block take effect when the threshold is crossed.
</Note>

When usage crosses the quota threshold, the policy engine applies the **quota block** action to subsequent requests. The request is rejected, and a `quota_block` event is written to the audit ledger.

<Warning>
  Once a tenant's quota is exhausted, all new requests from that tenant's applications are blocked until the quota is reset or raised. Make sure your applications handle quota-block error responses without surfacing raw error messages to end users.
</Warning>

## Showback views

The admin console breaks usage data down across three dimensions so you can understand consumption at the right level of detail:

  </Card>
  <Card title="By application" icon="grid-2">
    Drill into a tenant to see how usage is distributed across its registered applications. Useful when one application within a tenant is responsible for the majority of consumption.
  </Card>
  <Card title="By model" icon="cpu">

  </Card>

## How quota interacts with policy

Quota tracking and policy evaluation are tightly coupled. The quota percentage for a tenant is checked as part of policy evaluation on every request. If the tenant's usage has reached the threshold, the policy engine selects the quota block action before the request reaches Bedrock.

This means quota enforcement is:

- **Synchronous** — the block happens before the Bedrock call, so you are never billed for a request that was over quota.
- **Auditable** — every quota-block event is written to the audit ledger with the tenant ID, application, and timestamp.
- **Policy-driven** — the quota threshold is part of the tenant's policy configuration, not a separate system.

<Tip>
  Set quota thresholds conservatively during initial rollout. You can always raise a threshold for a healthy tenant, but a sudden spike in usage from an application bug can exhaust a generous quota before you notice.
</Tip>
