---
title: "Create and manage application API keys in Governix"
description: "Understand how Governix API keys are scoped to applications, where to generate them, how to use them at runtime, and how to keep them secure."
---

Governix uses API keys to attribute runtime requests and events to a specific application. When your code calls the Governix policy evaluation endpoint or sends usage events, it identifies itself with an application API key. This page explains how keys are scoped, how to create them in the admin console, and how to manage them safely.

## How API keys are scoped

API keys in Governix are **per application**, not per tenant. A single tenant can have multiple applications registered under it, and each application gets its own key. This lets you:

- Attribute usage, blocked events, and audit records to the specific part of your product that generated them
- Rotate a key for one application without affecting others
- Grant different parts of your team access to only the keys they need

<Note>
  Tenant identity is determined by the tenant ID you pass in your request payload or SDK call, not by the API key itself. The key identifies the application; the tenant ID identifies who the request is for.
</Note>

## Create an application and get its API key

<Steps>
  <Step title="Open the Applications page">
    Navigate to [app.governary.io/tenants](https://app.governary.io/tenants), select the tenant you want to register an application under, and open the **Applications** tab.
  </Step>
  <Step title="Create a new application">
    Click **Add application**, give it a name that reflects its role in your product (for example, `chat-api` or `document-search`), and save.
  </Step>
  <Step title="Copy the API key">
    After the application is created, the admin console displays the API key. Copy it immediately — Governix does not show the full key again after you leave this screen.

    <Warning>
      Treat your API key as a secret. Do not commit it to version control, log it, or include it in client-side code. Store it in your environment's secrets manager and inject it at runtime.
    </Warning>
  </Step>
  <Step title="Add the key to your integration">
    Pass the API key in the `Authorization` header or as the `api_key` field in your SDK configuration when calling Governix policy evaluation or event ingestion endpoints.
  </Step>
</Steps>

## Using API keys at runtime

Your application sends its API key with every call to Governix. The key is used to:

- **Authenticate the request** — Governix rejects calls with missing or invalid keys.
- **Attribute usage** — request counts, token usage, and estimated cost are recorded against the application that sent the key.
- **Populate audit records** — the application name appears on every audit ledger entry, making it easy to trace a request back to its source.

Pass the key in the `Authorization` header:

```http
Authorization: Bearer <your-api-key>
```

Or configure it once in your SDK:

```python
from governix import GovernixClient

client = GovernixClient(api_key="<your-api-key>")
```

## Rotate an API key

If a key is compromised or you want to rotate it as part of a regular security practice, generate a replacement in the admin console:

<Steps>
  <Step title="Open the application detail page">
    Navigate to the application whose key you want to rotate under [app.governary.io/tenants](https://app.governary.io/tenants).
  </Step>
  <Step title="Generate a new key">
    Click **Rotate API key**. The admin console generates a new key and immediately invalidates the old one.
  </Step>
  <Step title="Update your secrets manager">
    Replace the old key value in your environment's secrets manager and redeploy or reload any services that use it. Requests using the old key will be rejected as soon as rotation completes.
  </Step>
</Steps>

<Warning>
  Rotating a key invalidates the previous key instantly. Make sure you have updated all services that use the key before or immediately after rotating, to avoid authentication failures.
</Warning>

## API key management best practices

<AccordionGroup>
  <Accordion title="Store keys in a secrets manager">
    Use your cloud provider's secrets manager (such as AWS Secrets Manager or Parameter Store) to store Governix API keys. Avoid environment variable files checked into source control.
  </Accordion>
  <Accordion title="Use one application per logical service">
    Register a separate application for each distinct service or deployment that calls Governix. This keeps attribution clean and means a key rotation in one service does not affect others.
  </Accordion>
  <Accordion title="Rotate keys periodically">
    Even without a known compromise, rotating keys on a regular schedule (for example, every 90 days) reduces the window of exposure if a key is inadvertently leaked.
  </Accordion>
  <Accordion title="Audit key usage">
    Review the application-level usage breakdown in the admin console regularly. Unexpected spikes in request count from a single application can indicate a leaked or misused key.
  </Accordion>
</AccordionGroup>
