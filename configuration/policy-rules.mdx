---
title: "Configure per-tenant Bedrock policy rules in Governix"
description: "Define which knowledge bases and models a tenant can access, set citation requirements, configure fallback rules, and control runtime enforcement actions."
---

Governix lets you define policy rules for each tenant to control how they interact with your AWS Bedrock knowledge bases and models at runtime. This page covers how to configure those rules in the admin console at [app.governary.io/tenants](https://app.governary.io/tenants), what each field controls, and how Governix enforces policy decisions when a request arrives.

## What policy rules control

A tenant policy determines:

- **Knowledge base access** — which Bedrock Knowledge Base IDs the tenant is allowed to query
- **Model access** — which Bedrock model IDs or ARNs the tenant is permitted to invoke
- **Citation requirements** — whether responses must include source citations from the knowledge base
- **Fallback behavior** — which model to use if the requested model is not permitted
- **Quota enforcement** — the request and token thresholds at which Governix intervenes

<Note>
  In the current MVP, policy is tenant-scoped. Every application belonging to a tenant shares the same policy. Per-application policy overrides are not available yet.
</Note>

## Runtime enforcement actions

When Governix evaluates a request against a tenant's policy, it can take one of the following actions:

| Action | Description |
|---|---|
| `allow` | The request meets policy requirements and proceeds normally. |
| `deny` | The request is blocked because it references a disallowed knowledge base or model. |
| `force_filter` | The request is allowed but Governix applies a mandatory metadata or content filter before passing it to Bedrock. |
| `downgrade_model` | The requested model is not permitted; Governix substitutes the configured fallback model instead. |
| `quota_block` | The tenant has reached or exceeded their usage threshold; the request is blocked until the quota period resets. |

## Policy configuration fields

Use the following fields when creating or editing a tenant policy in the admin console.

<ParamField body="allowed_knowledge_bases" type="string[]">
  List of Bedrock Knowledge Base IDs the tenant is permitted to query. Requests referencing any ID not in this list trigger a `deny` action. Leave empty to deny all knowledge base access.
</ParamField>

<ParamField body="allowed_models" type="string[]">
  List of Bedrock model IDs or ARNs the tenant is permitted to invoke. Requests for a model outside this list trigger either a `deny` or a `downgrade_model` action, depending on whether a fallback model is configured.
</ParamField>

<ParamField body="require_citations" type="boolean" default="false">
  When set to `true`, Governix enforces that responses include citations from the retrieved knowledge base chunks. Use this for compliance or audit scenarios where source traceability is required.
</ParamField>

<ParamField body="fallback_model" type="string">
  The model ID or ARN to substitute when the tenant requests a model that is not in `allowed_models`. If this field is empty and the requested model is disallowed, Governix returns a `deny` action instead.
</ParamField>

<ParamField body="quota_threshold" type="object">
  Quota limits for the tenant. When exceeded, Governix applies a `quota_block` action. See [Set quota thresholds per tenant](/configuration/quota-thresholds) for the full field reference.
</ParamField>

## Effective policy visibility

The admin console shows the **effective policy** for each tenant — the resolved combination of all configured rules as Governix will enforce them at runtime. Use this view to confirm that the right knowledge bases and models are permitted before putting a tenant into production.

To view effective policy for a tenant:

<Steps>
  <Step title="Open the tenant list">
    Navigate to [app.governary.io/tenants](https://app.governary.io/tenants) and select the tenant you want to inspect.
  </Step>
  <Step title="Open the Policy tab">
    Select the **Policy** tab on the tenant detail page. You will see the full effective policy, including resolved knowledge base and model lists, citation requirements, fallback configuration, and quota settings.
  </Step>
  <Step title="Confirm the effective state">
    Verify that the **Effective KB policy** and **Effective model policy** sections reflect your intended configuration. Any mismatch here will be enforced at runtime.
  </Step>
</Steps>

<Tip>
  After changing a tenant's policy, review the effective policy view before routing live traffic. Runtime enforcement picks up changes immediately — there is no deployment step.
</Tip>
